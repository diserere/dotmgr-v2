# Pains & Expectations (исходный документ)

*Сырые заметки пользователя, с которых начался проект. Сохранены как исторический артефакт — основа для PRD.*

## Боли пользователя

- Моя основная рабочая среда — linux, в подавляющем большинстве случаев я использую один дистрибутив — Ubuntu, в версиях от 24.04 до актуальной 26.04, включая иногда не-LTS релизы. В основном — Ubuntu Desktop (Gnome), в части случаев — Server без GUI.
- Я имею как минимум 3 постоянно используемых рабочих места — 2 ноута и 1 постоянную виртуалку как staging стенд. Также я часто разворачиваю временные стенды, на которых тоже хочется иметь настроенное привычным образом окружение, или VPS на хостинге.
- Для меня актуальны 2 ситуации:
  - я часто добавляю useful скрипты или настройки в той системе, в которой работаю в данный момент, и потом приходится их копировать руками в остальные среды — например, руками добавляя нужные алиасы или настройки в конфиг git, или копируя конфиг в общую между ВМ и хостом папку.
  - я разворачиваю новую временную ВМ, и трачу достаточное время на установку нужных мне пакетов (которые примерно повторяются на всех машинах — git, curl, доп. пакеты для python типа python3-venv), затаскивание конфигов или алиасов, или ручную настройку окружения.

## Что хотелось бы иметь

- возможность централизованного хранения актуальных настроек с возможностью:
  - сохранения с любого хоста (сохранения актуальной локальной версии) в хранилище
  - обновления определённого конфига / группы конфигов на локальной машине из хранилища
- возможность на чистой системе с дефолтным набором окружения (ситуацию облегчает то, что можно ориентироваться практически на один дистрибутив, возможно, обобщив его до debian-based, но в целом даже это оверинжиниринг — для MVP 100%, так что можно ориентироваться на Ubuntu последних версий) с минимальными усилиями затащить эти настройки, желательно, с возможностью атомарного выбора — отдельно bash aliases, отдельно пакеты (вероятно, тоже по группам), конфиги (git, ssh, vim), отдельно конфиги для GUI утилит
- отдельной группой объектов для синхронизации являются секреты — пока могу вспомнить сразу разные API Key (в основном для LLM провайдеров — это всё free аккаунты, поэтому утечка не так страшна, но всё же), и ssh ключи. Но для них, как я понимаю, стоит искать какое-то отдельное решение.

## Как я это видел изначально

- для хранения мне показалось удобным использовать репо на GitHub
- сам интерфейс и архитектуру (когда я хотел просто написать его руками) я вначале вообще предполагал в виде простого bash скрипта, который последовательно предлагает скопировать в нужные места группы нужных данных, с дефолтным выбором y/N, чтобы я мог просто пустым вводом «прокликать» до нужного места, если мне надо скипнуть определённые фазы. Но когда я решил использовать для разработки агентов, то тут уже, конечно, можно было закладывать более развитый MVP сразу. Но мы как-то незаметно перешли к какому-то комбайну сразу на первом этапе :)
- почему bash — он есть сразу из коробки (и нативно поддерживает доступ к окружению, вызов команд и т.д. Ну и я его знаю). Потом в процессе гугления и обсуждения с агентом я узнал про существование stow — и это выглядит лучше, чем писать самому механизмы копирования и т.д. Выбор python был рекомендацией агента-архитектора, тут есть и плюсы, и минусы:
  - **плюс:** конечно, реализовать на нём достаточно сложную логику проще, чем на bash, не получив при этом страшную огромную портянку
  - **минусы:** для его разворачивания в системе надо доконфигурировать python — как минимум доставлять поддержку venv, и разворачивать его тоже в venv, поскольку наверняка понадобятся зависимости, которыми не хочется засорять хост систему. Т.е. в целом это сложнее по встраиванию в окружение (как я это вижу)
- первичную установку я предполагал по сценарию — я открываю в браузере страницу своего репо (поэтому предполагался публичный репо), и копирую со страницы однострочный скрипт-бутстраппер формата как сейчас используется для установки большинства проектов с GitHub — вида `wget <raw github url>/install.sh | bash`, который уже затаскивает первоначальный инсталлер. Или копирую эту строку с хост-системы в виртуалку или ssh-сессию. Потом, после установки git и gh, уже можно авторизоваться на GitHub, и работать со своими приватными репо (и этот сценарий может использовать любой другой юзер, который заинтересуется проектом). И вот в приватном репо уже можно хранить свои данные.
