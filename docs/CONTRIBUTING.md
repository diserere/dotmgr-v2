# Руководство по контрибьютингу

## Языковая политика

Рекомендуется использовать русский язык для документации, комментариев в коде и сообщений коммитов. Исключение составляют технические термины, названия API, библиотек и ключевые элементы инфраструктуры, которые следует сохранять на английском языке для однозначности.

## Git Workflow

*   **Запрет прямых коммитов в main:** Внесение любых изменений напрямую в ветку `main` запрещено. Все работы ведутся в feature-ветках.
*   **Ветвление:** Trunk-based development с использованием короткоживущих веток:
    *   `feat/<название>` — новая функциональность.
    *   `fix/<название>` — исправление ошибок.
    *   `docs/<название>` — изменения в документации.
    *   `refactor/<название>` — рефакторинг.
*   **Коммиты:** Используйте Conventional Commits (например, `feat(cli): добавить инициализацию`). Все коммиты, созданные при участии агента, должны содержать метаданные:
    ```
    Co-Authored-By: <Имя Агента> <email> [Role: <роль, например: Architect/Developer/QA>]
    ```
    *Примечание: `<email>` и `<Имя Агента>` должны соответствовать конкретному исполнителю. Пример:*
    ```
    Co-Authored-By: Claude <noreply@anthropic.com> [Role: Architect]
    ```
*   **Pull Requests (PR):** Открывайте PR на GitHub для фиксации истории обсуждений между агентами (использование комментариев). **Запрещено** выполнять слияние через GitHub UI.
*   **Merge:** Слияние выполняется **локально** в ветку `main`. Не используйте squash-merge, чтобы сохранить полную историю коммитов.
    ```bash
    git checkout main
    git merge <имя-ветки>
    git push
    ```
